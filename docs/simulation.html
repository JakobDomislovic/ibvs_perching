Running the simulation

One command → Gazebo, ArduPilot SITL, mavros, detector, controller.

Step 1

Prerequisites

The simulation is built on the LARICS uav_ros_simulation stack (ROS Noetic, Gazebo, ArduPilot SITL, ardupilot_gazebo). With that workspace in place, build this package once:

cd ~/uav_ws
catkin build ibvs_perching
source devel/setup.bash

Runtime dependencies are standard ROS packages: rospy, mavros, cv_bridge (with OpenCV's ArUco module), tf, and tmuxinator for the startup session.

Step 2

Start everything

cd ~/uav_ws/src/ibvs_perching/startup/sim_ibvs
./start.sh

start.sh links session.yml to .tmuxinator.yml and starts a tmux session named ibvs_perching_sim. Every pane exports UAV_NAMESPACE=red, so all topics live under /red/…. Give ArduPilot SITL ~30 seconds to boot before doing anything — its EKF needs a moment before it accepts arming.

GUID_OPTIONS must be 0

The startup sets the ArduPilot parameter GUID_OPTIONS to 0 before launching the controller. The LARICS kopterworx parameter file ships with GUID_OPTIONS = 8, which makes ArduPilot treat the attitude-setpoint thrust field as raw motor thrust instead of a climb rate — with that setting this controller commands a constant full climb and the vehicle flies away. The controller page explains the thrust semantics in detail.

Step 3

A tour of the tmux session

Move between windows with Ctrl+B then W (window list). Lines marked press ↑ are pre-typed into the pane's shell history — select the pane, press the up arrow, and hit Enter.

Window What runs in it
roscore roscore, ArduPilot SITL (sim_vehicle.launch, its console lives in the hidden ardupilot1 window), and mavros with use_sim_time. This window just needs to be alive — you rarely look at it.
gazebo Spawns the kopterworx quadrotor using the package's own camera URDF — the camera sits directly under the base looking straight down (the stock mount hangs 0.3 m low and would touch the tag at the final approach height).
visualization Pre-typed (press ↑): PlotJuggler with a ready-made layout for the command signals, RViz, and rqt_image_view on ibvs/debug_image — the detector's camera view with the detected marker drawn in. Open this one; it is the best way to watch the mission.
ibvs The heart of the session. Pane 1 sets GUID_OPTIONS 0 and launches ibvs_perching.launch (controller + ArUco detector + the tag model in Gazebo). Pane 2 has the keyboard joystick pre-typed (press ↑). The remaining panes echo mavros/state (mode + armed) and ibvs/state (the controller's state machine), plus a pre-typed rosbag recorder.
joystick The RC→joy bridge (rc_to_joy from uav_ros_general — the same node the real vehicle runs) and an echo of joy. When you fly with the keyboard you will see the stick values move here, exactly as they would with a real transmitter.
status Echoes of ibvs/target_point (what the vision module sees), mavros/setpoint_raw/attitude (what the controller commands), and odometry.
Step 4

Fly a mission

There are two ways to run the mission. The joystick mission is the main one — it rehearses exactly what will happen in the real world. The service mission is a fully autonomous fallback that needs no piloting at all.

A — Joystick mission (real-world rehearsal)

  1. Go to the ibvs window, select the keyboard pane, press ↑ and Enter to start keyboard_rc.py. Wait for “SYSID_MYGCS=1 — RC override enabled”.
  2. O arm → 2 ALT_HOLD → hold W to climb.
  3. Fly over the tag with ←↑↓→, watching ibvs/debug_image.
  4. Press I — the IBVS button. The next tag detection engages: the controller takes GUIDED_NOGPS itself and starts aligning.
  5. Take back control any time with 2; hand it back with G or press I again.
Full key reference

B — Service mission (fully autonomous)

Three std_srvs/Trigger services, called from any terminal:

# 1. GUIDED_NOGPS + arm + climb to 1 m, then hold
rosservice call /red/ibvs/takeoff

# 2. start servoing toward the tag
rosservice call /red/ibvs/start

# abort servoing, hold position
rosservice call /red/ibvs/stop

Takeoff climbs and holds — nothing else happens until you call ibvs/start. Watch ibvs/state: when it shows TAG_IN_SIGHT the detector sees the marker and ibvs/start will engage immediately.

What you should see

Follow ibvs/state in the ibvs window:

WAIT_ARM → CLIMB (service mission only) → HOVER / TAG_IN_SIGHT → ALIGN → ALIGNED

  • In ALIGN the vehicle first centers the marker laterally — descent is blocked until the marker is within 0.25 m of center (the descend gate), so the tag cannot escape the shrinking camera view.
  • Once centered it descends, regulating to a standoff of 0.5 m above the tag (target_z: -0.5).
  • ALIGNED means the error stayed below 15 cm for 2 s — the vehicle hovers centered over the marker.
  • If the detector loses the marker for 0.5 s the state drops to TAG_LOST and the vehicle holds position until the marker is seen again.
When things go wrong

Troubleshooting

Symptom Cause & fix
Does not take off (service mission) Did you call ibvs/takeoff? Arming alone never climbs. Also: ArduPilot auto-disarms after ~10 s on the ground, and right after SITL boot the EKF may reject arming — wait ~30 s and retry.
Flies away on takeoff GUID_OPTIONS is not 0 — the thrust field is being read as raw thrust. The startup sets it, but verify with rosrun mavros mavparam -n /red/mavros get GUID_OPTIONS.
Keyboard sticks do nothing ArduPilot only accepts RC overrides from the system id in SYSID_MYGCS. keyboard_rc.py sets it to 1 automatically — wait for the “RC override enabled” line before flying, and keep the keyboard pane focused (keys go to whichever pane is active).
Stuck in WAIT_ARM mavros/state shows mode ≠ GUIDED_NOGPS or armed: false. In the joystick mission this is normal until you press I and the tag is detected.
Stuck in TAG_LOST No fresh ibvs/target_point. Check ibvs/debug_image — is the marker in the camera view? Is the detector printing camera-info errors?
Pressed I, nothing happens Engagement needs three things: the vehicle armed, the controller running, and a fresh tag detection after the button press. Check ibvs/target_point is actually publishing.

More background on every failure mode is in the repository README.
