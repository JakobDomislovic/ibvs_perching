Code walkthrough

Every file in the package and what it is for

Nodes — scripts/

File What it does
ibvs_controller.py The controller — the heart of the package. A modal state machine (explained in depth here) that streams mavros/setpoint_raw/attitude at 30 Hz: body rates from a tilt cascade laterally, a climb-rate command vertically. Subscribes to ibvs/target_point (the vision interface), mavros/state and odometry; offers the ibvs/takeoff / ibvs/start / ibvs/stop services; publishes its state on ibvs/state (latched). Handles the mid-flight engagement (button-gated mode grab) and all the safety fallbacks: position hold, tag-loss hold, pilot-takeover one-shot.
aruco_detector.py The shipped vision module. Subscribes to camera/color/image_raw + camera/color/camera_info, detects a 4×4 ArUco marker (dictionary DICT_4X4_50, id 0) with OpenCV at 15 Hz, and publishes the marker center as a normalized image point + depth on ibvs/target_point. Draws the detection into ibvs/debug_image for the operator. Any node that publishes the same topic contract can replace it — the controller never knows the difference.
keyboard_rc.py Simulated RC transmitter (full guide). Turns keypresses into spring-loaded stick deflections on mavros/rc/override, switches flight modes and arms through the mavros services, and exposes the IBVS button (I / K → ibvs/start / ibvs/stop). Sets ArduPilot's SYSID_MYGCS = 1 at startup so the overrides are accepted. Simulation-only.
mock_ar_tag_publisher.py Camera-free vision mock. Computes where a tag at a known world position would appear in the down-facing camera from odometry alone, and publishes the same ibvs/target_point contract. Launch with use_mock_tag:=true to test the controller without rendering, detection, or lighting effects — useful for isolating control problems from vision problems.
rc_to_joy.py Fallback RC→joy bridge. A drop-in Python port of uav_ros_general's rc_to_joy node (median-filtered sticks, deadzones, the LARICS joy axis layout). The startups use the original stack node — this copy exists only so the package can run on a machine without uav_ros_stack. Not launched by default.

Launch files — launch/

ibvs_perching.launch — the main launch

Starts the controller and one vision module inside the UAV namespace, and (in simulation) spawns the tag model in Gazebo. No tracker, no control manager — the controller talks to mavros directly.

ArgumentDefaultMeaning
namespace$UAV_NAMESPACEUAV namespace (red in all startups).
configconfig/ibvs_params.yamlController parameter file — the real-world startup passes its own.
use_mock_tagfalsetrue replaces the ArUco detector with the odometry-based mock.

ar_tag_x / ar_tag_y0 / 0Where the tag is placed on the floor.

mavros_apm.launch — mavros bringup (real world)

A self-contained merge of uav_ros_general's apm2.launch + mavros_node.launch: starts mavros_node against the flight controller serial port (fcu_url), loading the package-local apm_config.yaml and the stock ArduPilot plugin lists. Exists so the aircraft needs only mavros itself, not the whole uav_ros_stack. (The simulation uses ardupilot_gazebo's own mavros launch instead.)

rc_to_joy.launch

Launches the fallback Python RC→joy bridge with a combined channel-mapping + deadzone config. Kept for stack-less machines; the startups launch the original uav_ros_general node.

Configuration — config/

FileWhat it configures
ibvs_params.yaml All controller parameters for simulation — engagement flags, climb-rate limits, cascade gains, PID gains, state-machine timing. Fully documented inline and in the parameter reference.
apm_config.yaml mavros plugin configuration for ArduPilot FCUs (copied from uav_ros_general so the package is self-contained). Used by mavros_apm.launch in the real world.
rc_to_joy.yaml Default RC channel mapping + stick deadzones for the fallback bridge.
plotjuggler_ibvs.xml Ready-made PlotJuggler layout: commanded body rates and climb rate, target-point error, and altitude — everything needed to judge a run at a glance.

Simulation assets — urdf/ & models/

FileWhat it is
kopterworx_downcam.urdf.xacro The kopterworx quadrotor with a custom down-facing camera mount: the camera sits directly under the base looking straight down, with image right = body forward. The stock down_facing_camera hangs 0.3 m below and 0.2 m to the side — it would physically touch the tag at the 0.5 m standoff, and its offset breaks the controller's camera-to-body assumption.
models/ar_tag/ Gazebo model of the target: a 40×40 cm plate carrying a 30 cm ArUco marker (DICT_4X4_50, id 0) as a texture. Spawned on the floor at the world origin by the main launch. The package exports its models/ directory to Gazebo so the texture resolves.

Startups — startup/

FolderWhat it starts
startup/sim_ibvs/ The simulation session — Gazebo + SITL + mavros + controller + joystick, toured window by window in the simulation guide.
startup/real_world/ The field session: mavros against the real FCU (mavros_apm.launch, serial port and RC mapping set per-aircraft in rw_setup.sh / custom_config/), the standard uav_ros_general RC→joy bridge (exactly as in the lab's other aircraft), a camera-driver slot, and the controller with real-world parameters. Same engagement workflow as simulation: fly manually, press the button, the next detection takes over. Includes package-local copies of the waitForRos-style shell helpers so the aircraft does not need the uav_ros_stack shell additions.

Finally, CMakeLists.txt / package.xml declare a plain Python catkin package — no compiled code, and only standard dependencies (rospy, mavros, cv_bridge, tf). The README carries the same story as these pages in text form, including the full troubleshooting section.
