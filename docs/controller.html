Docs · How it works

The IBVS controller

One Python node: a seven-state machine wrapped around two control loops.

Interface

The vision-module interface

The controller is deliberately target-agnostic: it knows nothing about ArUco markers. Any vision module can command it by publishing the point it wants centered in the camera image on ibvs/target_point (geometry_msgs/PointStamped):

FieldMeaning
point.x Normalized horizontal offset from the image center, (u − cx)/fx — positive right in the image.
point.y Normalized vertical offset from the image center, (v − cy)/fy — positive down in the image.
point.z Distance to the target along the optical axis in meters, or 0.0 if unknown. Descent only happens with a real depth; with z = 0 the controller centers the point while holding altitude (lateral gains are scaled by depth_guess).

Publishing on the topic at all means “target in sight” — freshness (within tag_timeout = 0.5 s) drives the TAG_IN_SIGHT / TAG_LOST distinction. The camera is assumed rigidly mounted looking straight down with image right = body forward (the kopterworx down-camera mount). Swap in your own detector by publishing this one topic — see the code walkthrough for the shipped ArUco implementation.

Behavior

The state machine

Global rules, always active: disarming or leaving GUIDED_NOGPS drops any state back to WAIT_ARM; ibvs/stop drops any flying state back to HOVER.

State Vertical command Lateral command
WAIT_ARM hover (0.5 = zero climb rate) level — setpoints are ignored anyway outside GUIDED_NOGPS
CLIMB climb_thrust (0.52) — a gentle constant climb is the takeoff position hold at the latched takeoff point
HOVER / TAG_IN_SIGHT hover position hold at the latched point
ALIGN / ALIGNED PID on target depth toward the 0.5 m standoff, gated by lateral centering IBVS cascade on the target point (below)
TAG_LOST hover position hold where the target was lost
ArduPilot semantics

“Thrust” is a climb rate — read this once

The controller streams mavros_msgs/AttitudeTarget messages with the IGNORE_ATTITUDE mask: body rates + a thrust value. In GUIDED_NOGPS, ArduPilot does not read that thrust field as motor thrust — it reads it as a climb-rate command:

thrust valuemeaning
0.0descend at maximum rate
0.5zero climb rate — hold altitude
1.0climb at maximum rate

This is why hover_thrust is 0.5, why takeoff requires a value above 0.5, and why the altitude never has to be "found" by a thrust estimator — ArduPilot's own vertical loop does the work.

The GUID_OPTIONS trap

Setting bit 3 of the ArduPilot parameter GUID_OPTIONS (value 8) switches the field to raw thrust interpretation — used by the uav_ros_stack MPC, and the default in the LARICS kopterworx parameter file. Under that interpretation this controller's hover command (0.5) is far above the real hover throttle (~0.29) and the vehicle climbs away at full rate. Both startups therefore force GUID_OPTIONS 0 before launching.

Control

The control laws

Lateral: a tilt cascade, not raw rates

Commanding a body rate proportional to the position error is unstable — the rate integrates into an unbounded tilt (flight-tested: it flips the vehicle). The working structure is a cascade with an attitude inner loop:

# outer loop: PID on the target position error -> a desired tilt (clamped small)
desired_tilt = PID_xy(target_error)                  # |tilt| ≤ max_tilt (0.15 rad)

# inner loop: attitude error -> body rate (runs in EVERY flying state)
body_rate = kp_att * (desired_tilt - current_tilt)   # |rate| ≤ max_body_rate

The PID's derivative term is fed the body velocity from odometry instead of a numerical derivative of the detection (for a stationary target, d(err)/dt = −velocity) — the same information with far less noise. Without this damping (kd) the lateral oscillation outruns the shrinking camera view during descent and the tag leaves the frame.

Position hold: level ≠ stationary

Outside of servoing, "just fly level" does not hover in place: the FCU's attitude trim bias (AHRS_TRIM) gives a constant lateral push, and velocity damping alone only caps the resulting drift at a terminal speed. The hold states therefore latch an (x, y) position and run a small position-P + velocity-damping loop on it through the same tilt cascade — that latched point is what HOVER, TAG_LOST and CLIMB keep.

Vertical: descend only while centered

In ALIGN, a PID on the target depth drives the climb-rate command toward the standoff (target_z: −0.5 → hover 0.5 m above the tag). A safety gate blocks any descent while the lateral error exceeds descend_xy_gate (0.25 m): descending off-center shrinks the camera's field of view faster than the lateral loop converges, and the tag falls out of frame — flight-tested the hard way.

Handover

Engagement — how the pilot hands over control

Two parameters (both enabled in the shipped configs) define the joystick-era workflow:

  • engage_on_target: true — the controller may seize GUIDED_NOGPS itself when a fresh target point arrives while the vehicle is armed and flown manually, and go straight to ALIGN (no takeoff, no position_hold handshake — the vehicle is already airborne).
  • engage_needs_start: true — engagement is button-gated: it only happens after ibvs/start is called (I on the keyboard joystick). Seeing the tag alone never grabs control.

The software mode switch is one-shot: the moment it fires, it disarms itself. After a safety-pilot takeover (RC mode switch), the controller cannot re-take the mode no matter what the vision module publishes — control returns only when the pilot flips back to GUIDED_NOGPS or presses the button again. If the engaging point goes stale before the mode switch completes, the controller simply holds position (TAG_LOST) — a button press plus one empty point is effectively a position-hold command.

Services

Service (std_srvs/Trigger)Effect
ibvs/takeoff Autonomous mission entry: switches to GUIDED_NOGPS, arms, climbs to takeoff_height, then holds. Refused while already flying.
ibvs/start Enables servoing and arms the one-shot engagement. Flying and the tag in sight → ALIGN; flying manually → the next detection engages.
ibvs/stop Stops servoing; the vehicle holds position (HOVER).
